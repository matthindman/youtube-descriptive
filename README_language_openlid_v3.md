# README: YouTube language classification with OpenLID-v3 on Databricks

## Purpose

`01_language_openlid_v3_databricks.py` performs first-cut language classification for the YouTube channel census. It uses OpenLID-v3 as the primary detector and keeps GlotLID scaffold code commented out for follow-up audits.

The notebook classifies text **segment by segment** and then aggregates segment predictions to a channel-level language label. This is intentionally not a single concatenated-text classifier: channel names, channel descriptions, video titles, video descriptions, and tags often use different languages.

## Files and default outputs

- Notebook: `01_language_openlid_v3_databricks.py`
- Segment-level predictions: `prod_tads.youtube.yt_lid_openlid_v3_segments`
- Channel-level vote table: `prod_tads.youtube.yt_lid_openlid_v3_channel_votes`
- Final channel-level language table: `prod_tads.youtube.yt_lid_openlid_v3_channels`

The notebook reads these source tables by default:

- `prod_tads.youtube.yt_sl_channels`
- `prod_tads.youtube.yt_sl_videos`

## OpenLID-v3 model setup

Default widgets:

```text
model_repo = HPLT/OpenLID-v3
model_filename = openlid-v3.bin
model_local_path = /dbfs/models/openlid_v3/openlid-v3.bin
download_model_if_missing = true
```

The OpenLID-v3 Hugging Face model page is internally inconsistent: the autogenerated snippet refers to `model.bin`, while the repository file list shows `openlid-v3.bin`. The notebook now tries the requested filename first, then `openlid-v3.bin`, then `model.bin`. Keep the default `openlid-v3.bin` unless the repository changes.

### Manual upload path

If the cluster has no internet access:

```bash
pip install huggingface-hub
huggingface-cli download HPLT/OpenLID-v3 openlid-v3.bin --local-dir ./openlid_v3

databricks fs mkdirs dbfs:/models/openlid_v3
databricks fs cp ./openlid_v3/openlid-v3.bin dbfs:/models/openlid_v3/openlid-v3.bin --overwrite
```

Then set:

```text
model_local_path = /dbfs/models/openlid_v3/openlid-v3.bin
download_model_if_missing = false
```

A Unity Catalog Volume path is also fine, for example:

```text
/Volumes/prod_tads/youtube/<volume_name>/models/openlid_v3/openlid-v3.bin
```

## Segment-level classification

The notebook auto-detects common columns. Override if needed:

```text
channel_id_column
video_id_column
channel_name_column
channel_description_column
video_title_column
video_description_column
video_tags_column
```

Each available text field is classified separately:

- `channel_name`
- `channel_description`
- `video_title`
- `video_description`
- `video_tags`

The video selection is deterministic. If a video timestamp/rank column exists, videos are sorted by that column and then by `video_id` to break ties.

## Channel-level aggregation

The notebook stores top-3 language predictions per segment. Channel-level aggregation now uses both top-1 and top-2 segment predictions:

- Top-1 segment prediction receives full weight.
- Top-2 segment prediction receives `secondary_label_vote_weight`, default `0.35`.
- Optional segment-length weighting is enabled by default with `use_lid_length_weighting=true`.

Default segment-type weights:

| Segment type | Weight |
|---|---:|
| channel_description | 3.00 |
| video_title | 2.00 |
| video_description | 1.00 |
| video_tags | 0.75 |
| channel_name | 0.50 |

This fixes a common bilingual-channel problem: if English is always the top-1 prediction but Spanish, Japanese, French, or another language is consistently top-2, the secondary language can still appear in the channel-level vote table.

## Output fields

The final channel table includes:

- `channel_id`
- `primary_language_label`, e.g. `eng_Latn`
- `primary_language_iso639_3`, e.g. `eng`
- `primary_language_script`, e.g. `Latn`
- `primary_language_confidence`
- `secondary_language_label`
- `secondary_language_iso639_3`
- `secondary_to_primary_score_ratio`
- `is_mixed_language_candidate`
- `language_status`
- `language_votes_json`

## Optional source-table update

The notebook does not modify `yt_sl_channels` by default. If you later enable:

```text
update_source_detected_language = true
```

then `source_update_format` controls what is written to `yt_sl_channels.detected_language`:

```text
source_update_format = label          # eng_Latn
source_update_format = iso639_3       # eng
source_update_format = scriptless_label
```

The default is `label`. Change it if downstream code expects bare language codes.

## GlotLID follow-up

GlotLID is kept as commented-out scaffold for a second-pass audit or low-confidence fallback. Prepare it manually with:

```bash
huggingface-cli download cis-lmu/glotlid model.bin --local-dir ./glotlid

databricks fs mkdirs dbfs:/models/glotlid
databricks fs cp ./glotlid/model.bin dbfs:/models/glotlid/model.bin --overwrite
```

Then set:

```text
enable_glotlid_fallback = true
glotlid_local_path = /dbfs/models/glotlid/model.bin
```

For the first-cut analysis, leave GlotLID disabled.

## Validation notes

Create a stratified validation set from:

- high-confidence major-language channels;
- low-confidence channels;
- mixed-language candidates;
- non-Latin scripts;
- channels where source `language_code` disagrees with OpenLID-v3;
- sparse-text channels.

Do not use YouTube `defaultLanguage` as the only ground truth. It is useful as an audit field, but it is sparse and usually refers to metadata language rather than spoken channel language.

## Cautions

- Do not classify one concatenated text blob as the main method.
- Do not overwrite source tables until validation is complete.
- Review OpenLID-v3’s GPL-3.0 license implications before distributing the model binary.
- Collapse model outputs to a project-level language taxonomy before publication.
